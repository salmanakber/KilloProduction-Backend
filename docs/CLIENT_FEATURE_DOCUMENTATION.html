Platform Overview (Detailed)

SuperKillo is a unified super-platform where commerce, mobility, service providers, and finance operate on one backend. Customers can buy products, request services, book riders, transfer money, and track every lifecycle from request to settlement, while administrators control KYC, security, operations, and policy.

Core Pillars

  • Commerce modules: Pharmacy, Food, Grocery, Auto Parts, Supplier sourcing
  • Service module: Mechanic quotes, offers, job execution, and approval flow
  • Mobility module: Ride booking, courier dispatch, rider acceptance, live tracking
  • Financial rails: wallet, transfers, withdrawals, gateway processing, refunds
  • Admin governance: KYC, compliance, commissions, security, reporting, HR

Shared Infrastructure

  • AI layer for recommendation, virtual assistant, and intelligent drafting
  • Socket layer for dispatch, chat, status updates, and presence
  • Order lifecycle engine for split orders, pickup, delivery, and completion
  • Commission and tax handling with module-specific payout calculations
  • Role-based access model for admin features and operational safety
  • Notification infrastructure: in-app, email, SMS, and mobile push (device token registry) with module-aware routing

Note on messaging: Marketing campaigns drive growth and conversions; system notices communicate operational truth (incidents, maintenance, policy); both coexist with transactional messages (orders, payouts, OTPs) delivered through the same technical channels where appropriate.

1) Pharmacy Module (Deep Explanation)

Pharmacy is built as both a healthcare-assist module and a medicine commerce engine. It links AI symptom support, prescription processing, pharmacy matching, chat/approval workflow, checkout, rider logistics, and health continuity records.

Customer Experience Features

  • Virtual doctor flow: customer can describe symptoms with text, voice, or image and receive medically guided suggestions.
  • Prescription upload and shortlist: prescriptions are analyzed and transformed into structured medicine suggestions for purchase.
  • Nearby pharmacy intelligence: location-aware pharmacy listing with stock/availability and delivery suitability checks.
  • Pharmacy chat handoff: customer can continue clarification in chat when AI suggests uncertain or partially available items.
  • Purchase flow: address, payment, and confirmation handled in-app with module-aware checkout behavior.

Backend/Operations Features

  • Single-store first matching strategy: attempts complete fulfillment from one pharmacy before split sourcing.
  • Prescription queue: pharmacies receive queued requests with status control for review, acceptance, and follow-up.
  • Top-selling and search endpoints: powers medicine discovery, ranking, and fast purchase conversion.
  • Payment/settlement hooks: provisional vendor crediting and final settlement on completion events.
  • Refund integration: supports post-order refund handling including reverse pickup logistics when required.

Health Integration

  • Medicine reminders and vitals: pharmacy behavior can connect with ongoing health management data.
  • Doctor visit and lab context: health timeline can support better medication continuity decisions.
Workflow: symptom/prescription input -> AI extraction and medicine shortlist -> pharmacy review queue + chat -> customer payment -> rider pickup and delivery -> completion settlement and notifications.

2) Supplier (Wholesaler) Module

Supplier module enables pharmacies/vendors to source medicines and products from wholesalers with quote negotiation, courier-enabled delivery, payment/commission handling, and inventory import after delivery.

Supplier Lifecycle Features

  • Wholesaler onboarding and KYC: admin can create, verify, approve, suspend, and govern wholesaler accounts.
  • Wholesaler catalog management: wholesalers maintain products and medicine lists for B2B sourcing.
  • Supplier discovery: pharmacies can browse wholesalers and evaluate supplier detail before ordering.

Quote and Order Features

  • Quote request creation: pharmacy/vendor sends supplier order request with needed items and terms.
  • Supplier response options: wholesaler can accept, reject, or counter offer with changed conditions.
  • Terms confirmation: pharmacy confirms final terms before acceptance and payment progression.
  • Delivery coordination: courier booking can be created for wholesaler-to-pharmacy movement.
  • QR-secured handover: supplier order QR can be generated and verified for controlled pickup/delivery.
  • Inventory import: delivered supplier items can be imported directly into pharmacy inventory records.

Financial and Governance Features

  • Commission and distance calculators: backend supports pricing/fee calculations tied to order acceptance.
  • Notification bridge: supplier-specific notifications for quote updates and operational events.
  • Admin eligibility tooling: admin can inspect supplier-pharmacy compatibility and control module exposure.
Workflow: admin-approved wholesaler publishes catalog -> pharmacy sends quote -> wholesaler responds -> terms are confirmed -> supplier order accepted and paid -> courier pickup/delivery with QR verification -> pharmacy imports delivered items into stock.

3) Auto Parts Module (Detailed)

Auto Parts is a marketplace module where customers request parts, vendors send custom offers, and the platform supports part-only or part-plus-service journeys with rider/courier logistics and strong completion controls.

Core Features

  • AI-assisted request drafting: backend AI helps convert user problem descriptions into structured part requests.
  • Vendor scoring and matching: intelligent matching by location, compatibility, budget, and vendor performance.
  • Custom offer workflow: vendors submit detailed offers (condition, warranty, timeline, price).
  • Offer acceptance engine: customer acceptance transitions request/order states and prepares payment/delivery.
  • Delivery quote support: route and courier estimation provided before dispatch execution.
  • Handover verification: pickup and delivery safeguards using secure code/QR checks.

Integrated Modes

  • Part-only mode: customer buys part and receives rider/courier delivery.
  • Part + mechanic mode: accepted part offer can enter mechanic bid/service flow before final completion.
Workflow: customer raises request -> vendors send custom offers -> customer accepts -> delivery quote and payment -> courier/rider dispatch -> handover verification -> completion and settlement.

4) Mechanic Module (Standalone + Integrated)

Mechanic module is a complete service system, not just a small extension. It supports direct quote flow, service request statuses, chat/communication, additional-part requests, approval-based payout release, and admin KYC/governance.

Customer and Mechanic Features

  • Direct quote request flow: customer can request quote from mechanics without a vendor-part chain.
  • Quote lifecycle: quote pending -> submitted -> accepted/rejected with controlled transitions.
  • Service request lifecycle: pending, quoted, accepted, in-progress, completed, cancelled, rejected states.
  • Mechanic offer flow: for integrated jobs, mechanics can submit offers and customer accepts one winner.
  • Additional parts request: mechanic can request extra parts during active service.
  • Chat and notifications: dedicated mechanic chats and notification events for all milestones.

Earnings and Control Features

  • Escrow-like pending credits: mechanic earnings can remain pending until approval/completion trigger.
  • Approval-gated release: customer approval or completion finalization releases pending wallet credits.
  • Distance-based pickup pricing: configurable price-per-km for pickup-related mechanic jobs.
  • Rating system: post-service customer ratings update mechanic quality metrics.

Admin Mechanic Controls

  • KYC decisions: approve/reject with reasons and audit trail updates.
  • Module stats: active mechanics, job volumes, revenue indicators, rating summaries.
  • Settings control: platform-level mechanic pickup pricing and policy surfaces.
Workflow: quote/offer creation -> customer acceptance -> service execution updates -> completion request -> customer approval -> payout release and rating.

5) Food Module (Detailed)

Food module is optimized for timing-sensitive fulfillment. It combines AI assistance, menu/catalog controls, delayed-dispatch strategy based on prep time, promotions, and POS-ready API integrations.

  • AI food assistant: smart suggestions, meal plan conversations, and guided selection.
  • Preparation-aware dispatch: rider notification can wait until food is closer to ready state to reduce idle time and improve freshness.
  • ETA prediction: combines route intelligence and prep estimates for more realistic delivery windows.
  • POS integration: vendor APIs and keys for external system synchronization.
  • Promotion and loyalty support: promo validation, usage tracking, and rewards impact at checkout.
  • Multi-order decomposition: can split complex orders and coordinate pickup/delivery sequence.
Workflow: discover items -> optional AI help -> checkout with promos -> prep-time aware rider dispatch -> live tracking -> completion and accounting updates.

6) Grocery Module (Detailed)

Grocery module provides product-centric shopping with strong inventory visibility and fast rider interconnection.

  • AI pantry assistant: weekly planning and smart reorder guidance.
  • Catalog controls: categories, stock flags, featured products, and pricing behavior for stores.
  • Fast dispatch model: grocery checkout commonly enters immediate rider request flow.
  • Promotions and campaign visibility: offers, coupons, and campaign discovery on customer home surfaces.
  • Multi-pickup support: parent/child order handling for complex fulfillment structures.
Workflow: browse stores -> AI assist or direct add-to-cart -> checkout -> immediate rider assignment -> tracking and delivery completion.

7) Standalone Riding Module

Riding is a separate operational module that powers passenger/courier booking, dispatch waves, rider acceptance, OTP safety checks, and lifecycle-linked earnings.

  • Booking orchestration: creates ride/courier bookings with payment and promo validation.
  • Eligibility-aware request feed: only suitable riders receive each request based on service type and context.
  • Atomic accept flow: request acceptance updates assignment and earning records safely.
  • OTP start verification: trip starts only after secure OTP confirmation at pickup.
  • Live operations: sockets for location, status, and request lifecycle broadcast.

8) Customer Super App Module

Customer app layer unifies all module journeys behind one navigation, one account identity, and common payment/wallet/loyalty systems.

  • Unified navigation: customer can move between pharmacy, food, grocery, auto-parts, riding, and money transfer features.
  • Unified cart and checkout behavior: module-specific handoff with consistent UX and account context.
  • Address, payment methods, and wallet controls: reusable account capabilities across all modules.
  • Loyalty and rewards: point accrual/redemption and customer history view.
  • Smart shopping tools: AI suggestions and saved plan logic to improve repeat behavior.
  • Authentication experience: phone/email login with optional OTP verification paths; push tokens refresh on supported devices after login so notices and orders reach the right handset.
  • Chat & real-time: module-threaded conversations (e.g. pharmacy, mechanic) synchronized with order and service state.

9) Money Transfer Module

Money transfer module enables secure P2P movement with compliance checks, gateway abstraction, payout processing, and dedicated admin oversight.

  • Transfer creation: sender to registered receiver with limits, validations, and fee computation.
  • Bank verification constraints: transfer and withdrawal flows enforce bank-account validity.
  • Gateway flexibility: Stripe/Paystack flow selection driven by runtime configuration.
  • Payout execution: receiver can withdraw to bank and track payout status.
  • History and monitoring: detailed transaction timelines in mobile and admin.

10) Bonus and Loyalty Systems

  • Rider peak bonus engine: creates demand-based challenges and tracks completion progress in real time.
  • Challenge acceptance flow: riders can accept active incentive tasks and receive event-based updates.
  • Customer loyalty engine: points, caps, expiries, rewards, redemption logs, and module-specific earning behavior.
  • Admin visibility: rider bonus analytics and platform tuning connect incentives to operational outcomes.

11) Marketing Management & Automation (Growth Suite)

This is a separate growth layer from day-to-day promo codes and from operational system notices. It is designed for planned campaigns, measurable engagement, and repeatable audience outreach across modules.

Campaigns & Channels

  • Multi-channel outreach: combine push, email, and SMS according to campaign strategy and compliance.
  • Scheduling & timezones: start/end windows, frequencies (one-off, recurring patterns), and calendar-aware execution.
  • Audience targeting: filter by user types, platform modules, segments, and geographic or cohort signals where configured.
  • Content & CTAs: titles, body copy, images, deep links, and call-to-action text for measurable clicks.
  • A/B testing: variant splits for message optimization on supported campaign types.

Segments & Automation

  • Segments: reusable audience groups for campaigns and reporting (member counts and engagement quality).
  • Automation rules: trigger-based journeys with execution history, success/failure monitoring, and iteration.
  • Analytics loop: delivery, opens, clicks, conversions, revenue attribution, and health indicators to refine future campaigns.
How it differs: Promo codes are typically short redemption mechanics at checkout. Marketing campaigns orchestrate outreach and measurement. System notices (next section) are operational broadcasts, not promotions.

12) System Notices & Operational Broadcasts

When the platform must communicate maintenance windows, incidents, policy updates, or safety guidance, administrators use a dedicated system notice workflow—distinct from marketing campaigns and promo spam risk.

  • Channels: mobile push (Expo push tokens stored per user), email, SMS, or in-app inbox-only notices.
  • Lifecycle: draft, schedule for a future time, send immediately, or rely on background workers for due schedules.
  • Audience: target by user roles (e.g. customers, vendors, riders) with counts computed from live user data.
  • Scale: delivery runs in controlled batches so large audiences do not overload a single job.
  • Governance: sent notices are locked; drafts and scheduled items can be edited or removed before dispatch.

13) Special Offers, Vendor Flash Events & Approval Gates

Beyond generic promo codes, the platform supports curated promotional programs and vendor-initiated flash or mystery offers that may require administrative approval before customers see them.

  • Special offers workspace: create and monitor platform-wide or targeted offers with an approval path for pending items.
  • Vendor offers: mystery or flash-style promotions submitted by vendors route through a pending queue for compliance and quality.
  • Why it exists: protects brand trust—promotional intensity is gated while still enabling fast merchant creativity.

14) Medicine Catalog, Taxonomy & Data Quality

Pharmacy intelligence depends on clean catalog data. Administration tools keep medicines, conditions, and sourcing metadata aligned with customer discovery and AI flows.

  • Master medicine directory: maintain the canonical medicine list powering search, prescriptions, and fulfillment.
  • Wholesaler suggestions: review catalog additions or corrections proposed through B2B sourcing workflows.
  • Illness categories: structure symptom and condition browsing so AI and search route users sensibly.
  • Medicine origins / sourcing metadata: enrich listings for compliance, filtering, and supplier transparency where configured.

15) Developer, POS & Partner Integrations

Food service and omni-channel vendors often rely on external POS or partner systems. The platform exposes developer-grade controls so integrations stay secure over time.

  • POS & partner APIs: provision credentials for trusted integrations.
  • Key lifecycle: rotate or revoke keys without downtime surprises; reduce blast radius if a secret leaks.
  • Vendor-side settings: food module settings align external menus and operational behavior with in-app ordering.
  • Optional runtime configuration: Firebase/push and messaging-related settings can be centralized for consistent mobile behavior.

16) HR, Employees & Internal Workforce

The admin panel includes internal workforce controls separate from marketplace users (customers, riders, vendors).

  • HR dashboard: leave requests, attendance-oriented stats, and staffing summaries for operations leadership.
  • Employee directory: manage internal admin and operational staff records alongside HR workflows.
  • Granular admin access: feature grants and legacy permission mappings ensure each admin sees only approved areas (payments, marketing, complaints, notifications, etc.).

17) Money Transfer — Administrative Oversight

Customer-facing transfers (section 9) are mirrored by an admin console for monitoring, configuration, and payout governance.

  • Dashboard: snapshot of volume, risk indicators, and operational posture.
  • Transactions: searchable ledger of movements for support and reconciliation.
  • Payouts: oversee withdrawal and settlement pipes tied to user bank destinations.
  • Configuration: tune limits, fees, provider selection, and policy knobs without redeploying apps.
  • Bank verification / transfer KYC: administrative review surfaces tied to verified funding endpoints.

18) Support, FAQs & Customer Experience Operations

  • Complaints / tickets: structured intake, threading, and resolution workflows for production incidents and disputes.
  • FAQ management: publish and refine self-service answers that reduce repeat contacts.
  • Operational awareness: pending-count cues on navigation highlight queues needing attention (approvals, tickets, etc.).

19) Configuration, Compliance & Observability

  • Global settings: session timeouts, notification defaults, loyalty parameters, and cross-module toggles.
  • Security center: security events with investigate/resolve flows and configurable defensive policies.
  • Audit logs: trace administrative actions for accountability and regulator-ready narratives.
  • Email templates: maintain transactional and system templates with variable placeholders.
  • AI configuration: central governance of model/prompt defaults powering assistants across modules.
  • Ride types & rider tooling: configure ride categories and analyze peak bonus performance.
  • Vendor performance: operational analytics to identify strong or struggling merchants.
  • Cross-module categories: taxonomy maintenance where product families span storefront experiences.

20) Detailed Backend Admin Panel (Deep Domain Map)

Admin panel is a full command center with domain-specific workbenches. It is not only CRUD; it contains policy engines, approval flows, financial controls, incident handling, and reporting/export.

Admin Domains and What They Control

The sidebar mirrors operational reality: module-specific workbenches sit next to horizontal platforms (payments, marketing, HR, support). Below is a consolidated map aligned with what administrators actually use day to day.

  • Dashboard: KPIs, module summaries, revenue and operational snapshots.
  • User management: account lifecycle, activation, roles, and profile governance across customers and supply-side users.
  • KYC & verification: unified queue for vendor/rider/mechanic verification with approve/reject, reasons, and audit trail.
  • Orders: cross-module order intervention—cancellation, completion assists, refund pathways—with action logging.
  • Payments hub: payment records, wallet movements, retries/cancellations, refunds workspace, withdrawal approvals, and consolidated stats.
  • Vendor management by module: Auto Parts, Pharmacy, Food (restaurants + settings), Grocery, plus shared categories taxonomy.
  • Riders: fleet oversight, peak bonus analytics, and ride type configuration.
  • Mechanics: directory and verification aligned with the mechanic service module.
  • Promo codes: redemption mechanics, limits, and lifecycle distinct from full marketing campaigns.
  • Commission management: rules, activation, module economics, platform fees.
  • Medicine administration: medicines master list, wholesaler catalog suggestions, illness categories, medicine origins.
  • Marketing management: growth dashboard—campaigns, segments, automation, engagement and conversion analytics.
  • Money transfer (admin): dashboard, transactions, payouts, configuration, bank verification.
  • Special offers & vendor offers: platform offers plus pending mystery/flash vendor submissions.
  • Notifications (system notices): operational broadcasts—not promotional marketing—with scheduling and channel selection.
  • Support: complaints/tickets and FAQ content management.
  • Reports & analytics exports: module and financial reporting pipelines.
  • Developer: POS/partner API credentials, rotation and revocation.
  • Settings cluster: general platform settings, security incidents, audit logs, email templates, AI configuration.
  • HR & employees: internal workforce—staff HR views and employee records.
  • Wholesalers: B2B supplier account oversight tied to pharmacy sourcing (alongside supplier workflows in section 2).
Admin operating pattern: monitor -> decide (approve/reject/adjust policy) -> execute operational action -> audit and report outcomes.

Interconnected Architecture Highlights

  • Shared rider network: Pharmacy, Food, Grocery, Auto Parts, and supplier-linked deliveries can feed dispatch.
  • Shared real-time backbone: socket events synchronize dispatch, chat, status, and presence across modules.
  • Shared settlement logic: wallet/payout/commission flows are connected with module-specific release rules.
  • Shared governance: admin controls unify security, compliance, and operational policy across the full platform.
  • Shared AI strategy: AI assistants are integrated into module experiences and centrally governed via admin config.
  • Growth vs operations: marketing automation and promo tooling layer on top of the same user graph without confusing shoppers—system notices stay factual and separable from promotions.
  • Data & catalog spine: medicine taxonomy, categories, and vendor catalogs feed both customer discovery and admin reporting.

Result: SuperKillo operates as one interconnected digital operating system for commerce, service fulfillment, logistics, and finance.
